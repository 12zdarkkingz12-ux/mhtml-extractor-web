# MHTML Extractor — Web

أداة ويب تستخرج الصور المضمّنة من ملفات MHTML/MHT وتحزمها في ZIP منظم،
مع رفع مجزّأ (chunked) وشريط تقدم حقيقي ومعالجة بالخلفية، مناسبة للنشر على
Render (الخطة المجانية).

## هيكل المشروع

```
app/
  app.py             # سيرفر Flask: رفع مجزأ + معالجة بالخلفية + polling
  core.py            # منطق الاستخراج: تحليل MHTML، التحقق من الصور، بناء ZIP
  templates/index.html
  requirements.txt
  Procfile           # أمر تشغيل gunicorn على Render
```

## كيف يعمل الاستخراج (core.py)

1. **parse_mhtml**: يقرأ الملف ويفكك أجزاءه (HTML + الصور المضمّنة).
2. **collect_ordered_images**: يطابق كل `<img src="cid:...">` أو `data:` أو اسم
   ملف مع الجزء المقابل له، بترتيب ظهورها في الصفحة، ثم يضيف أي صور متبقية
   لم تُذكر صراحة كـ fallback.
3. **التحقق من سلامة كل صورة (is_valid_image_bytes)**: قبل إضافة أي صورة
   للأرشيف، يتم التأكد من الـ magic bytes الحقيقية للملف (PNG/JPEG/GIF/WEBP/...).
   أي جزء يدّعي أنه صورة لكن بياناته تالفة أو فارغة يُستبعد ولا يدخل الـ ZIP.
4. **تقرير شفاف (stats)**: كل عملية استخراج ترجع إحصائية واضحة:
   - `img_tags_found`: عدد وسوم `<img>` في الصفحة
   - `matched_from_html`: كم منها تمت مطابقته فعلياً بصورة
   - `unmatched_img_tags`: كم وسم ما لقي له صورة (صور خارجية لم تُضمّن)
   - `fallback_parts_added`: صور أضيفت احتياطياً لم تكن مذكورة في HTML
   - `invalid_images_skipped`: صور فشلت بفحص الـ magic bytes وتم تجاهلها
5. هذا التقرير يظهر للمستخدم في الواجهة بعد كل عملية استخراج، فما فيه "نجاح
   صامت" قد يخفي مشاكل.

## لماذا رفع مجزّأ (chunked) بدل رفع دفعة واحدة؟

- Render Free له حدود على مهلة الطلب (request timeout) وحجم الذاكرة.
- رفع ملف 200MB دفعة واحدة قد يعلّق المتصفح بدون أي مؤشر تقدم، وقد يفشل
  بصمت لو انقطع الاتصال.
- بالتقسيم لقطع 5MB، الواجهة تعرض شريط تقدم حقيقي لكل قطعة، وأي فشل بقطعة
  يظهر برسالة واضحة بدل ما يفشل الرفع كامل من الصفر.
- المعالجة (الاستخراج) تتم في **thread بالخلفية** بعد اكتمال الرفع، فطلب
  `/api/upload/complete` يرجع فوراً، والواجهة تتابع التقدم عبر
  `/api/status/<job_id>` (polling كل 800ms) لحد ما تخلص.

## ملاحظة مهمة عن الذاكرة (200MB حد أقصى)

Render Free عادة 512MB RAM. قراءة ملف 200MB + فك تشفير base64 (يكبر
~33%) + بناء ZIP ممكن يقترب من الحد. النظام الحالي يكتب القطع مباشرة
على القرص (مو بالذاكرة) أثناء الرفع، لكن خطوة التحليل نفسها (`BytesParser`)
تحمّل الملف كامل بالذاكرة دفعة وحدة. لو لاحظت تعليق/تعطل السيرفر مع ملفات
قريبة من 200MB، الحل المقترح:
- تقليل الحد الأقصى الفعلي إلى ~120-150MB كهامش أمان، أو
- الترقية لخطة مدفوعة بذاكرة أكبر، أو
- تعديل `core.py` ليعالج الـ MIME parts بشكل تدريجي (streaming parser) بدل
  `BytesParser` الكامل — تحسين مستقبلي إذا واجهتك مشاكل فعلية.

## التشغيل محلياً

```bash
cd app
pip install -r requirements.txt
python app.py
# افتح http://127.0.0.1:5000
```

## النشر على Render

1. ادفع المشروع لمستودع GitHub.
2. على Render: New → Web Service → اختر المستودع.
3. Root Directory: `app` (أو عدّل المسار حسب هيكلتك على GitHub).
4. Build Command: `pip install -r requirements.txt`
5. Start Command: (موجود بالـ Procfile تلقائياً، أو يدوياً)
   ```
   gunicorn app:app --worker-class gthread --workers 1 --threads 4 --timeout 120 --bind 0.0.0.0:$PORT
   ```
6. **مهم جداً: لازم `--workers 1`**. حالة المهام (jobs) محفوظة بالذاكرة
   داخل العملية (in-memory dict)، فلو شغّلت أكثر من worker، كل worker
   بيكون عنده نسخته الخاصة من الـ jobs وما راح يلقى المهمة اللي بدأت
   بـ worker ثاني → أخطاء "غير موجود". لو احتجت أكثر من worker مستقبلاً
   لازم تنقل حالة المهام لمكان مشترك (Redis مثلاً) بدل الذاكرة.
7. اختر الخطة Free، وانشر.

## تحسينات مستقبلية مقترحة

- نقل حالة المهام لـ Redis لو احتجت أكثر من worker/instance.
- إضافة حد زمني (TTL) أوضح لحذف الملفات المرفوعة فوراً بعد فشل/نجاح
  المعالجة بدل الانتظار ساعة كاملة.
- دعم استئناف الرفع (resume) لو انقطع الاتصال أثناء رفع ملف كبير.
- معالجة streaming لملفات MHTML الضخمة بدل تحميلها كاملة بالذاكرة.
